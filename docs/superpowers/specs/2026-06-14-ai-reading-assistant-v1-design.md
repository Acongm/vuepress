# AI 阅读助手 v1 全链路优化设计

## 决策

采用方案 A：静态缓存自举 + 内容分析哈希。一次性发布独立的 AI v1 构建与对话链路；旧接口继续兼容，但前端只接入 v1，避免在旧协议上反复渐进调整。

本设计替代 `2026-06-14-ai-reading-assistant-optimization-design.md` 中的覆盖式抽屉和旧接口渐进方案。

## 已确认问题

- GitHub Actions 最新 master 构建当前成功，约 3–4 分钟；历史全量 AI 分析构建达到 30–38 分钟。
- workflow 缓存 key 包含全部 Markdown 的总哈希，每次新增或编辑文章都会生成新 key。
- 线上缓存回填只恢复摘要，不恢复 manifest；Vercel 冷构建无法判断哪些文章未修改。
- 构建缓存命中不稳定，不能保证第二次相同构建零 AI 调用。
- GitHub Actions 使用 Node 20，已出现 2026 年 6 月 16 日切换 Node 24 的弃用警告。
- 打开 AI 助手后，部分文章没有展示预分析内容；页面打开与运行时摘要兜底之间的边界不清晰。
- PC 端原有潜入文档分栏符合阅读习惯，但当前小屏降级布局异常。
- 当前文章、本模块、联网检索被实现为硬模式控件，不符合自然对话体验。

## 成功标准

1. 连续两次对相同提交构建，第二次 AI API 调用数为 `0`。
2. 仅修改一个可分析 Markdown 后，AI API 调用数为 `1`。
3. 未修改文章的分析结果在 GitHub Actions 和 Vercel 冷构建中均可复用。
4. 用户打开页面或展开助手时不调用任何 AI 后端接口。
5. 默认展示构建阶段生成的文章摘要；没有可用摘要时展示明确的静态空状态，不自动实时生成。
6. 只有用户发送对话才调用 `/api/ai/v1/*` 并消耗运行时 token。
7. PC 保持潜入文档分栏；平板使用覆盖侧栏；手机使用底部全屏面板。
8. 320、375、390、768、1024、1440 像素宽度下无横向溢出、遮挡或不可操作控件。
9. GitHub Pages 与 Vercel 测试部署成功，构建过程使用 Node 24。

## 构建分析 v1

### 单一缓存快照

`summaries-v1.json` 是可部署、可恢复的完整快照：

```json
{
  "version": 1,
  "analysis": {
    "model": "deepseek-v4-pro",
    "promptVersion": "summary-v1",
    "extractVersion": "markdown-v1"
  },
  "files": {
    "/react/react16.md": {
      "sourceHash": "sha256:...",
      "analysisHash": "sha256:...",
      "status": "success",
      "summary": {},
      "processedAt": "..."
    }
  },
  "stats": {}
}
```

不再依赖单独且可能丢失的 manifest。每个文件条目同时持有摘要与失效判定元数据。

### 哈希与失效

- `sourceHash`：规范化后的 Markdown 可分析内容哈希。
- `analysisHash`：`sourceHash + model + promptVersion + extractVersion` 的哈希。
- 仅以下情况调用 AI：没有条目、`analysisHash` 变化、上次状态不是 success。
- Git diff 只用于输出诊断信息，不参与最终复用判断。
- 删除文章时删除对应缓存条目。
- README、过短内容和明确排除目录生成确定性的本地状态，不调用 AI。

### 缓存恢复

恢复顺序：

1. 当前工作区 `.cache/ai-summaries-v1.json`。
2. GitHub Actions cache 恢复的快照。
3. 上次成功部署的 `/summaries-v1.json`。
4. 全新空快照。

远程恢复拿到的单文件元数据足以直接判断复用，不再需要恢复另一份 manifest。

### 构建输出

每次输出：总文件数、复用数、待分析数、短文跳过数、失败数、AI 调用数、命中率、分析耗时。`--dry-run` 只打印计划，绝不调用 AI。

没有 `AI_API_KEY` 时保留已恢复摘要；不能用空文件覆盖现有可用缓存。

## CI/CD v1

### GitHub Pages

- workflow 和 setup-node 使用 Node 24。
- 使用 `npm ci`，锁文件必须保持同步。
- cache 使用稳定版本前缀和分支回退，不使用全部 Markdown 总哈希作为唯一 key。
- AI 分析、VuePress 构建和 Pages 发布是可识别的独立步骤。
- AI 分析失败时构建失败，不能以 `|| echo` 掩盖真实错误。
- 部署使用官方 Pages artifact 流程或已验证的 gh-pages 流程，避免重复发布链路。

### Vercel

- Vercel 构建调用同一 `build:ai:v1`，行为与 GitHub Actions 一致。
- 构建前从上次成功站点恢复 `summaries-v1.json`。
- 测试环境记录构建总耗时和 AI 调用数。
- 目标：无文章变化时不发生 AI 调用；VuePress 静态编译时间作为主要耗时。

## AI API v1

新增并冻结：

- `POST /api/ai/v1/chat`
- `POST /api/ai/v1/chat/stream`

v1 一次性提供：

- 服务端所有 system prompt。
- 最近 6 轮对话裁剪。
- 文章或模块上下文字符预算。
- `max_tokens` 上限。
- SSE 流式回答、停止生成、usage 和 sources。
- 联网检索仅在用户明确选择对应快捷 Tag 时启用。

旧 `/api/ai/chat` 保持兼容，但新前端不再使用它。

## 页面零调用原则

页面加载、摘要预取、助手展开和作用域选择均只访问站点静态资源：

- `/summaries-v1.json`
- `/module-index.json`
- 当前页面 DOM

不得调用 `/api/ai/*`。无构建摘要时显示“此文章暂无预分析内容”，并允许用户直接提问；发送问题后才调用 v1 Chat。

## Chat 交互

### 快捷 Tag

输入框上方展示可点击 Tag：

- `当前文章`
- `本模块`
- `联网检索`
- 根据文章摘要生成的 2–4 个建议问题

点击 Tag 后向输入框插入可编辑的自然语言片段，例如：

- `结合当前文章，`
- `结合本模块相关文档，`
- `联网检索最新资料后，`

Tag 仅辅助组织问题，不立即发送、不切换隐藏模式。请求参数从最终输入和已选择 Tag 派生，用户可组合多个 Tag。

### 消息与输入

- 默认首条内容是构建摘要卡片，不伪装成运行时回答。
- Enter 发送，Shift+Enter 换行。
- 流式生成时发送按钮变为停止按钮。
- 支持重试、重新生成、复制与清空当前文章会话。
- 会话本地保存并限制大小；模型请求仅包含最近 6 轮。

## 响应式布局

### PC：≥ 1180px

- 保留潜入文档方式：文章容器平滑收窄，助手在同一 `.page` 网格内出现。
- 助手宽度在 400–520px 之间响应变化。
- 目录空间不足时收起右侧 TOC，但正文保持可读最小宽度。

### 平板：768–1179px

- 助手成为右侧覆盖面板，不改变正文排版。
- 宽度 `min(520px, 88vw)`，带轻遮罩；点击遮罩或 Escape 收起。

### 手机：< 768px

- 从底部弹出，先显示约 76vh；可继续展开为全屏。
- 输入区固定在可视区域底部，适配 `env(safe-area-inset-bottom)` 和虚拟键盘。
- 320px 宽度下 Tag 横向滚动，按钮不挤压，消息不溢出。

### 动画

- PC 组合使用网格宽度、位移与透明度，缓动 `cubic-bezier(.22,1,.36,1)`。
- 平板从右侧滑入；手机从底部弹出。
- 消息进入使用短距离上移和淡入，不对长文本逐字模拟。
- `prefers-reduced-motion` 下关闭位移动画并缩短淡入。

## 摘要覆盖与无解析文章

- 构建报告列出未生成摘要的 Markdown 路径和原因。
- 失败条目不标记为 success，下次构建自动重试。
- 增加覆盖率检查；非排除文章缺少 success 摘要时，测试构建失败。
- 线上助手严格按路径规范化匹配 `.html`、目录首页和 `.md` 缓存键。

## 测试与验收

### 自动测试

- 构建缓存：首次 N 次调用、第二次 0 次、改单篇 1 次、失败重试、删除清理、无 Key 保留缓存。
- API v1：历史裁剪、上下文预算、流式事件顺序、停止、usage、旧 API 兼容。
- 前端：SSE 分块、Tag 插入与组合、页面打开零 Chat/summary API 请求、路径匹配。
- 构建：`npm ci`、AI 数据测试、VuePress build、bundle smoke。

### 浏览器矩阵

- 320×568、375×812、390×844。
- 768×1024、1024×768。
- 1440×900、1920×1080。

每个尺寸验证：展开/收起动画、摘要展示、Tag、输入、停止、滚动、横向溢出和控制台错误。

### 测试环境

- GitHub Pages 测试分支完整通过。
- Vercel Preview 完整通过。
- 连续两次部署证明第二次 AI 调用数为 0。
- 修改一个测试 Markdown 后证明 AI 调用数为 1。
- 记录部署 URL、构建耗时和缓存统计后才允许完成任务。
