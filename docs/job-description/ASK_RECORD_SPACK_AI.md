## B. 口语版（面试现场口播）

> 口语版原则：不“总结优势”，只讲事实与决策过程，让面试官从你的表达里感受到能力。

### 1）自我介绍（口语）

**面试官：请做一个自我介绍。**

你好，我叫彭聪，毕业于江西财经大学计算机应用技术专业。我从 2016 年开始做前端，2019 年开始做前端负责人相关的工作。

我这几年主要做的是中后台复杂系统的重构和长期治理：例如用户中心入口系统的多端统一和性能链路优化；售后工单系统的重构、工程规范从 0-1 落地、以及后续的组件沉淀和低代码提效；另外也做过跨系统公告插件 SDK、DevOps 平台、以及可视化报表的前后端一体化交付。

您想先听哪个方向？我可以从用户中心这个项目开始。

### 2）用户中心重构（口语）

**面试官：介绍一下用户中心重构。**

用户中心是登录入口，用户量和访问量都很大。当时最核心的目标不是“换新框架”，而是把风险控制住、把维护成本降下来、把首屏体验拉起来。

所以我们做的是增量演进：保留 Vue2 + webpack 主体不动，减少大迁移风险；用响应式把 PC/移动两套代码合并成一套；同时用一套可度量的性能治理，把包体积、关键资源优先级、第三方脚本、图片字体这些“最影响首屏”的点逐个解决。最后效果就是维护成本明显下降，登录转化也有提升。

**面试官：打包优化怎么做？**

这个问题的根因其实是老项目依赖越积越多、引入方式不按需、静态资源没人治理，包自然会变大。登录页包一大，弱网下用户就会等很久。

我当时先用构建产物分析把大头找出来（polyfill、重复依赖、字体/图片、首屏不需要的模块），然后按“polyfill 按需、拆包、长缓存、静态资源治理”把体积压下来，同时每次改动都用产物 diff + 线上 RUM 采样对比确认效果。

**面试官：更快的加载怎么做？**

加载慢通常不是一个点，是网络、资源优先级、渲染阻塞一起造成的。我会先用 Network/Performance 定位主要矛盾：是连接耗时、还是关键资源没优先级、还是主线程脚本太重。对应地做 HTTP/2/TLS 1.3 这类网络侧协同，关键域名 preconnect、关键资源 preload、脚本 defer、三方脚本延后并加超时降级，再配上压缩和缓存。

**面试官：JS 顺序怎么管？**

我会把脚本分层：业务关键脚本用 defer 保证顺序并且不阻塞解析；独立三方脚本要么 async，要么延后，还要加失败降级；不影响首屏的模块用动态 import。核心目标就是：首屏不要被脚本卡住，交互不要被长任务卡住。

**面试官：WebP 怎么兜底？**

WebP 的收益很大，但兼容风险也很大。我的做法是 picture/srcset 做前端兜底，同时构建产出双份资源并配好缓存策略；如果遇到特殊 WebView，就做能力探测缓存结果；最后用图片失败率和 LCP 波动去监控，保证升级不引入可用性问题。

### 3）RMS 重构（口语）

**面试官：说说 RMS 系统重构。**

RMS 是全球售后工单系统，页面多、权限复杂。我们当时的出发点是：让团队在复杂度持续上升时还能稳定交付。

我做的事情是分阶段：先把工程基础打牢（TypeScript + ESLint 规则从 0-1 建起来，把变量/类型/import/异步处理这些容易出问题的点工具化卡住）；然后把经常复用的模块沉淀成业务组件，并把组件逐步做成可配置的物料；在这个基础上再去做低代码的贡献与规模化使用，最后才能真正提效。

**面试官：ESLint 为什么要从 0-1 做？怎么做？**

因为规模一上来，靠人肉 review 是守不住的。我们遇到的问题很具体：变量声明不规范、类型定义不合理、Promise 不处理、import 顺序乱、模块边界不清晰，这些都会变成线上 bug 或维护成本。

所以我从 0-1 把 parser、插件、规则体系搭起来，重点规则直接 error 卡住；同时把自动修复放到提交前，把校验放到 CI，让规范变成流水线能力。存量代码不一刀切，先卡新增，再分目录逐步收紧。

**面试官：组件高可用怎么做？**

核心就是把边界条件收敛：统一 loading/empty/error，关键区域错误隔离；再把组件 API 做契约化，类型别乱；最后把组件的配置项、默认值、事件和校验整理出来，为后续物料化和低代码接入打基础。

**面试官：低代码怎么落地？**

我不直接一上来就推全量低代码，而是先找“高重复、结构稳定”的表单/配置类页面试点。组件这边先贡献物料，让 schema 能够稳定驱动；运行态渲染器负责递归渲染、表达式、事件分发；schema 做版本化和迁移，表达式能力做白名单与权限控制。这样才能在稳定性不下降的前提下把提效做出来。

### 4）公告插件 / DevOps / XDR（口语简版）

公告插件的目标就是跨系统低成本接入，所以我提供了 CDN/ESM/Web Component 多形态，SDK 本身追求极小体积，并且把 init/destroy、错误隔离、资源隔离和监控补齐，保证不会影响宿主系统。

DevOps 主要是把部署这类高风险重复劳动流水线化：镜像规范、分阶段流水线、安全扫描、按环境发布、健康检查和回滚。

XDR 报表定时任务本质是异步任务系统：幂等/锁、任务状态持久化、队列 worker、并发控制、监控告警，保证“按时生成、失败可追溯可补跑”。
